## envoy

스티오(Istio) 서비스 메시 환경에서 엔보이는 각 서비스(애플리케이션)의 앞단에 사이드카(Sidecar) 형태로 배치되어,
모든 네트워크 트래픽을 중재하고 제어하는 역할

1. 사이드카 패턴의 핵심 애플리케이션 코드 수정 없이 네트워크 기능을 추가하기 위해, 서비스 바로 옆에 붙어서 동작합니다. 
   비유하자면 오토바이(서비스) 옆에 붙은 보조석(엔보이)과 같습니다.

2. 주요 기능

    트래픽 관리: 로드 밸런싱, HTTP/gRPC 라우팅, 서킷 브레이커 등을 수행합니다.

    보안: 서비스 간 통신 시 TLS 암호화(mTLS)를 자동으로 처리합니다.

    관찰 가능성(Observability): 통과하는 모든 트래픽에 대한 로그, 메트릭, 추적(Tracing) 데이터를 생성하여 시스템 상태를 한눈에 보게 해줍니다.

    회복 탄력성: 네트워크 타임아웃 처리나 재시도(Retry) 로직을 대신 관리합니다.


## Virtual Service

이스티오(Istio)에서 가상 서비스는 서비스 메시 내로 들어오거나 서비스 간에 흐르는 트래픽을 어디로, 어떻게 보낼지 결정하는 핵심 역할

1. 트래픽 제어 및 라우팅 (Traffic Routing) 가장 기본적인 역할로, 요청이 들어왔을 때 특정 버전의 서비스로 트래픽을 보냅니다.

    카나리 배포(Canary Release): 트래픽의 10%는 신규 버전(v2)으로, 90%는 기존 버전(v1)으로 보낼 수 있습니다.

    A/B 테스트: HTTP 헤더 정보를 보고 특정 사용자(예: 아이폰 사용자)만 특정 서버로 접속하게 할 수 있습니다.

2. 서비스 주소의 추상화 클라이언트는 실제 서비스의 개별 IP나 복잡한 주소를 알 필요가 없습니다. 버추얼 서비스에 정의된 가상 호스트 이름을 호출하면, 이스티오가 이를 해석해 실제 목적지로 연결해 줍니다.

3. 장애 복원력 (Resilience) 네트워크 불안정성에 대비한 규칙을 설정할 수 있습니다.

    리트라이(Retries): 호출 실패 시 몇 번이나 다시 시도할지 설정합니다.

    타임아웃(Timeouts): 응답이 일정 시간 이상 걸리면 연결을 끊도록 설정합니다.

4. 트래픽 조작

    Fault Injection: 시스템 테스트를 위해 의도적으로 에러를 발생시키거나 지연(Latency)을 주입할 수 있습니다.

    
    spec:
        hosts:
        - reviews # 서비스 이름
        http:
        - match:
            - headers:
                end-user:
                exact: jason # 사용자가 jason이면
            route:
            - destination:
                host: reviews
                subset: v2 # v2 버전으로 보내라
        - route:
            - destination:
                host: reviews
                subset: v1 # 나머지는 v1으로 보내라


## DestinationRule

① 서비스의 그룹화 (Subsets)
    하나의 서비스 안에 여러 버전(v1, v2, v3)의 파드들이 섞여 있을 때, 이를 논리적으로 분리합니다.

    예: "라벨이 version: v1인 파드들을 v1이라는 이름의 그룹(subset)으로 묶는다."

② 부하 분산 정책 설정 (Load Balancing)

    특정 그룹 내에서 트래픽을 어떤 방식으로 나눌지 결정합니다.

    Random: 무작위 선택

    Round Robin: 순차적 선택

    Least Conn: 연결이 가장 적은 곳 선택

③ 접속 차단 및 회로 차단 (Circuit Breaking)
    특정 파드가 비정상적일 때(예: 500 에러 지속 발생) 해당 파드를 일정 시간 동안 호출 대상에서 제외하여 시스템 전체의 마비를 막습니다.

④ 보안 설정 (TLS)
    서비스 간 통신 시 TLS 암호화를 사용할지, 혹은 일반 텍스트로 통신할지 결정합니다.

2. VirtualService와의 관계

    트래픽 흐름을 보면 두 리소스의 차이가 명확해집니다.

    VirtualService (길 찾기): "사용자 트래픽의 80%는 reviews 서비스의 v1 서브셋으로 보내고, 20%는 v2로 보내라."

    DestinationRule (세부 정책): "v1 서브셋은 라벨이 version: v1인 파드들을 의미하며, 이들 사이에서는 Random 방식으로 부하를 분산해라."
